# 리플래닝 v7.4 — 전면 리뷰 진단과 R1~R4 로드맵

> 상태: **확정** (2026-07-23 사용자 결정 반영). 이 문서가 v7.4 이후 라운드의 기준 기획서다.
> 관계: `회원가입-Google-도입-기획서.md`(R2 편성으로 채택), `산책길-대시보드-기획서.md`(테마 결정 대기), `프로세스-단순화-제안.md`(항목별 상태 이관)와 함께 읽는다.

## 0. 서비스 정의 (모든 판단의 기준)

**자기 발견 기반 비전보드 서비스**: "나는 어떤 사람인지", "어떤 삶을 살고 싶은지"를 발견하고 그 모습을 생생하게 시각화하도록 돕는다.

- **시간이 걸리는 것 자체는 결함이 아니라 본질.** 혼자 하든 다른 도구를 쓰든 비전보드는 원래 오래 걸리는 작업이다.
- 허들 축소의 대상은 **"깊이"가 아니라 "마찰"**(불명확한 질문, 중복 입력, 막힘, 끊긴 흐름)이다.

## 1. 진단 요약

### 잘된 점 (유지·강화)
- **철학의 일관성**: 정체성 우선 → 언어 → 이미지 순서가 페이지 구조(section→scene→scenes)로 그대로 구현됨.
- **경로 유연성**: 질문 먼저/사진 먼저 양경로, 상태 기반 라우팅(`lib/sectionRoute.ts`), 자유 순서.
- **방어 코드 성숙**: quota boolean, 손상 백업, fail-open 검증, 압축 파이프라인, v7.4 감사(apiGuard).
- **일기 = 핵심 차별점**: "미래의 하루"를 글로 물성화. Peak-End 보상 구조.
- **Unsplash 파이프라인 완성도**: 검색 + AI 힌트 + 큐레이션 + download 핑 준수.

### 아쉬운 점 (이번 리플래닝의 대상)
1. 첫 가치까지 거리 + 세션 분할 부재 — "한 섹션 = 한 세션" 미명시, 첫 "내 보드다" 체감 늦음
2. 카피 과잉·중복 — `/scene` 안내 5겹, "순서는 네 마음" 3~4회 반복
3. 온보딩 설득 2스텝의 수동성 (읽기만 하는 카피)
4. 스토리 문체 이질 — 섹션 일기(일기체 450~650자) vs 최종 스토리(요약체 300자)
5. 과금 누수 — 일기 재생성 무제한, 미사용 SDK
6. 영속성 부재 — 브라우저 데이터 삭제 = 전부 유실
7. 구조 부채 — `/scene` vs `/scenes` 혼동, `/review` 역할 중첩, 레거시 필드

## 2. 6섹션 구조 재진단 — "지금 상태가 최선인가" (v7.4 신규)

### 사실 확인
- 6섹션(나·건강·관계·일·성장·돈·생활·공간·환경)은 **오프라인 종이 워크숍 원본을 그대로 디지털화**한 것. "왜 6개인가" 재검토 기록 없음 (원 설계: "막막함을 6개로 쪼개 진입 장벽을 낮춘다", `Playground/docs/비전보드_플로우_v0.6.md`).
- **6/6 게이트는 단 3곳**: ① 대시보드 /review 버튼(6섹션 텍스트 완료), ② 대시보드 /finish 직행 버튼(전 섹션 completed), ③ 콜라주 /finish 버튼+최종 스토리(`completedCount===6`).
- `/finish` 페이지 자체엔 게이트가 없고, `/api/story`·`/api/summarize`는 이미 빈 섹션을 자동 필터링하며, **콜라주·폰/PC 배경화면 저장은 이미 사진 1장부터 동작**한다(레이아웃도 항목 수에 동적 적응).
- → **결론: 6/6 허들은 데이터 구조 문제가 아니라 CTA 조건 3곳 + 하드코딩 카피 몇 곳의 문제.** 시스템은 이미 부분 상태를 관용하는데, 표시 계층이 부분 상태를 "미완성"으로 프레이밍하는 것이 마찰의 실체다.

### 구조 판정: 6섹션 프레임 유지 (사용자 확정)
분야별 분할은 생각을 쉽게 하고, 비전보드는 원래 천천히 만드는 것. 회피 영역(돈·건강 등)이야말로 자기발견 가치가 가장 큰 칸이므로, 전원에게 6칸을 주는 원본 워크숍의 설계가 서비스 철학에 맞다.

### 허들의 실체 3지점과 해체 전략
| 지점 | 마찰 | 해법 | 편성 |
|---|---|---|---|
| 시작점 | "6개나 해야 해?" | "먼저 마음 가는 3칸부터 시작해도 돼" 권장 카피 + 분모 6 재프레이밍(초반 구간 분모 노출 완화) | **R1 (카피)** |
| 중간 | "아직 보여줄 게 없어" | 부분 가치 노출: "1칸만 채워도 네 보드야 — 배경화면 저장도 돼" | **R1 (카피)** + R3 (첫 리빌 넛지) |
| 종점 | "피날레(최종 스토리)는 6/6 뒤에만" | **"첫 보드" 마일스톤**: 3섹션 completed 시 /finish·최종 스토리 조기 개방. 6/6은 "완전한 보드"로 상향 목표화. 보드가 자라면 "이야기 다시 써줄까?" 재생성 넛지 | **v7.8** (구 R2, §5) |

- R2 구현 메모: 게이트 ②③만 완화(`completedCount >= 3`), 게이트 ①(/review)은 유지. `api/story` 프롬프트의 "6개 섹션" 하드코딩 제거. `storyWrittenAtCount?` 옵셔널 필드(schemaVersion 불변). 임계값 3은 잠정 — R1 슬롯 유예의 착지를 보고 확정.
- 코어 경로 저장 버전(`focusSectionIds?` + 인트로 3칸 선택 UI + `getRecommendedSection` focus 우선)도 R2에서 검토. R1은 카피만(라이트).

### 폐기 결정: 선택형 섹션 (사용자 확정)
"사용자가 보드에 담을 섹션을 고르고 나머지는 제외" 안은 **도입하지 않는다.**
1. 회피 영역을 처음부터 빼게 해주면 자기발견 철학을 스스로 훼손한다 (불편해서 피하는 칸이 발견 가치가 가장 큰 칸).
2. `SECTION_IDS`·미니보드 6셀·ProcessBar·콜라주 키·API 프롬프트까지 전 화면 재작업 수준의 대공사.
3. "먼저 마음 가는 3칸부터" 코어 경로가 같은 심리적 이득(전부 다 안 해도 된다)을 프레임 훼손 없이 제공한다.

## 3. 질문 아키텍처 재진단 (`lib/questions.ts` 전수 정독)

**4+1 아크(지금→원해→더 들여다보기→방향 키워드→장면)는 유지. 질문 수 불변.** 심화는 기존 helpQuestions 3개로 충분. 개별 정비 4건만:
1. **S3(관계) 'current' 성격 불일치** — 라벨은 "지금"인데 질문은 미래·유산("10년 뒤 어떤 사람으로 기억됐으면") → 현재 인식 질문으로 교체, 유산 질문은 helpQuestions로 이동 검토
2. **S6(공간) 'feeling'↔장면 중복** — 둘 다 하루 흐름을 물음 → feeling을 "그 공간이 주는 감각" 각도로
3. **S2(건강) 'feeling'이 want와 중복** — 욕구 나열 반복 → 정서/상태(몸 감각·기분) 각도로
4. **S4(일)·S5(돈) 'current'의 결핍 전제** — "제일 아쉬운 게"는 만족 상태 사용자를 소외 → 양방향 문안으로

추가: **"나중에 답할게요" 유예**(PRD v0.2 D4/D6 설계 실현, keyword 슬롯은 유예 불가 — 장면·finish 재료) + **세션 분할 카피**("오늘은 여기까지도 충분해")로 한 섹션 = 한 호흡 공식화.

## 4. 질문별 결론 (2026-07 전면 리뷰 4문항)

- **Q1 가입·서버 저장**: 가입 강제 비추천 확정. 게스트 완주 유지, 배경화면 저장 직후(C) 주 유도 + 대시보드 보조 배너(B). Auth.js(NextAuth v5)+Google / Neon Postgres 무료티어 / Vercel Blob. **R2 편성** (기획서 상태 "보류"→"채택").
- **Q2 유저 여정**: 아래 로드맵 R1~R4.
- **Q3 Pinterest/Unsplash**: **Pinterest API 연동 폐기 확정** (심사·OAuth·약관 부담 대비 가치 낮음). 대신 직접 업로드 강화: 다중 파일 업로드(R1) → 드래그앤드롭·클립보드 붙여넣기(R3). Unsplash 현행 유지.
- **Q4 하루 일기**: **유지 (핵심 차별점)**. 조임 3가지 — ① 재생성 합산 2회 제한 후 "직접 수정하기" 권유(R1), ② 최종 스토리 문체 통일(R3, 일기 합성 방식 추천), ③ 일기 모아보기(R3).

## 5. 로드맵

- **R1 (v7.4, 완료)**: 기획서 갱신 + 무비용 퀵윈 — 재생성 2회 제한, 온보딩 몰입 강화(스킵 추가 안 함), 완료 시트 연속 흐름+세션 분할 카피, pathSheet 선택 기억, 카피 중복 정리+구조 카피(부분 가치·코어 경로 라이트·개화 카피 제거), 다중 업로드, 미사용 SDK 제거, **LLM 전면 무료화(OpenAI→Gemini 무료 티어, 폴백 Groq)**, 질문 정비 4건, 슬롯 유예.
- **v7.5~v7.6 (완료)**: 산책길 대시보드(WalkPathMap)+콜라주 폰/PC 2탭(v7.5), 숲 테마·시작 어포던스·예시 3세트 로테이션·씬 브리지(v7.6).
- **v7.7 (실사용 피드백 R2, 이번 라운드)**: 미니보드 숲 정합(흰 폴라로이드 잔존 제거), 상태 아이콘 재매핑(💬 답하는 중/📷 사진 차례 — "다음 행동" 지시)+데스크톱 title 툴팁, 산책길 완료 구간 실선("걸어온 길"), /collage 코치마크 토리 발화자 승격, 질문 예시 3→5세트(전환기형·관계공동체형 추가), walkCaption "차분히 한 칸" 넛지. 상세: `피드백-라운드-v77.md`.
- **v7.8 (첫 보드 마일스톤 — 구 R2에서 분리)**: 3섹션 completed 시 /finish·최종 스토리 조기 개방(§2 허들 종점 해체), 보드가 자라면 "이야기 다시 써줄까?" 재생성 넛지, 6/6 "완전한 보드" 상향 목표화. **로그인 의존이 전혀 없어(로컬 status·completedAt만으로 판정) 계정 R2보다 먼저 단독 착수 가능** — 지속성 장치 피드백(v7.7 라운드)과 직결되어 앞당김.
- **R2 (계정·영속성)**: Google 로그인 첫 슬라이스(Auth.js+Google, Neon 업서트, base64→Blob, C지점 유도+B 배너, 병합 시트, 계정 시트, 개인정보처리방침). "첫 보드" 마일스톤은 v7.8로 분리.
- **R3 (여정·결과물)**: 1섹션 완성 시 collage 첫 리빌 넛지, 최종 스토리 문체 통일(일기 합성), 일기 모아보기, 드래그앤드롭·붙여넣기 업로드. ※ 산책길 대시보드 테마는 별도 결정.
- **R4+ (백로그)**: 다중 보드(연도별), 마이페이지 2차, `/scene`→`/day` URL 정리(리다이렉트 동반), `/review` 축소·대시보드 통합 검토. (선택형 섹션은 백로그 아님 — §2 폐기.)
