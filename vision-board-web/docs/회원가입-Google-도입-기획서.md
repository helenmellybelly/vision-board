> 배경: 현재 100% localStorage(`vision-board-data`, schemaVersion 4). 기기를 바꾸거나 브라우저 데이터를 지우면 보드가 사라진다. 비전보드는 한 번에 끝나지 않는 작업이라 이어하기·재방문 여정이 필요해질 수 있다.

## 1. 목표와 원칙

- **목표**: (a) 보드 유실 방지(서버 백업), (b) 기기 간 이어하기, (c) 재방문 여정의 시작점.
- **원칙: 가입 없는 경험을 그대로 유지한다.** 게스트로 온보딩→보드 완성→배경화면 저장까지 전부 가능. 가입은 "잃지 않기 위한 선택"이지 관문이 아니다. (온보딩 앞 가입 강제는 이 서비스의 감정 곡선 — 토리와의 친밀감 형성 — 을 깨뜨림)

## 2. 가입 유도 시점 — 후보 비교와 추천

| 후보 | 시점 | 근거 | 리스크 |
|---|---|---|---|
| A | 온보딩 직후 **강제** | 이탈 전 확보 | 아직 잃을 것이 없어 동기 약함, 첫인상 훼손 → **강제는 계속 안 함** |
| A′ | **온보딩 종료 선택 화면** ⭐채택 | 시작 시점에 "Google로 시작 / 게스트로 시작"을 대등하게 제시 — 강제가 아니라 선택 | 게스트 유의점을 정확히 알려야 함(§2-1) |
| B | **첫 사진이 보드에 담긴 직후** ⭐주 유도 | "잃으면 아까운 것"이 처음 생기는 순간 | 작업 흐름 중간 방해 → 닫기 가능한 시트로, 1회만 |
| C | 배경화면 저장 직후 (보조 재유도) | 감정 정점 | 완주 전 유실은 못 막음 → 보조로 강등 |
| D | 재방문 2회차 진입 시 | 재사용 의사가 검증된 유저 | 그 사이 유실 위험 → 사용 안 함 |

**v7.5 확정 조합**:
1. **온보딩 종료 선택 화면(A′)** — 온보딩 3스텝이 끝나면 대시보드 진입 전에 "Google로 시작하기 / 게스트로 시작하기" 2버튼. 게스트 선택도 1탭으로 동등하게 가능(하드 게이트 아님).
2. **B = 주 유도 소프트 게이트** — 게스트가 첫 사진을 담은 직후 시트: "이 보드를 저장하고 이어가려면 Google로 로그인해둬" + 닫기 가능. 닫으면 게스트로 계속, 이후 대시보드 상단 1줄 배너로 재유도(재노출 7일 간격).
3. **C = 보조 재유도** — 배경화면 저장 완료 시트에 "이 보드, 계속 간직할래? — Google로 저장해두기" 1줄 유지.
어떤 지점에서도 **모달 강제 없음**(§1 원칙 유지 — 선택 화면이지 관문이 아니다).

### 2-1. 온보딩 종료 선택 화면 — 게스트 유의 카피

- 버튼: `Google로 시작하기` / `게스트로 시작하기`.
- 게스트 선택 시(또는 버튼 아래 상시) 유의 문구 — **"캐시" 같은 기술 어휘 금지**, 사용자 언어로:
  > **"지금은 이 기기 브라우저에만 저장돼. 기기를 바꾸거나 브라우저 데이터를 지우면 사라질 수 있어. 언제든 Google로 로그인하면 안전하게 보관돼."**
- 게스트로 시작한 유저가 나중에 로그인하면 §5 병합 규칙으로 로컬 보드를 그대로 승계 — "게스트로 시작해도 손해 없음"을 카피에 반영.

## 3. 재방문 여정 (로그인 유저)

1. 진입(`/`) → 세션 확인 → 서버 보드 존재 시 `/dashboard` 직행 + 복귀 인사(기존 `showReturnGreeting` 재사용).
2. **기기 A에서 만들고 기기 B에서 로그인**: 서버 보드를 내려받아 localStorage에 주입(아래 병합 규칙) → 이어서 작업.
3. 미완성 보드: 추천 카드가 그대로 다음 행동 안내 (현 구조 재사용 — 여정 로직 신규 개발 불필요).

## 4. 마이페이지 범위 (1차는 최소)

- 1차: 프로필(이름 = board.userName과 동기) · 내 보드 1개 · 로그아웃 · 계정 삭제. 별도 페이지 대신 대시보드 헤더 시트로 시작.
- 2차(보류): 보드 여러 개(연도별), 배경화면 저장 이력, 일기 모아보기.

## 5. 데이터 마이그레이션·병합 전략

- **모델**: `users(id, google_sub, email, name, marketing_consent boolean default false, marketing_consent_at timestamptz, created_at timestamptz default now())` + `boards(user_id, data jsonb, schema_version, updated_at)` — 보드 전체를 jsonb 1행으로(현 스키마 그대로). 정규화는 보드 다중화 시점에. `marketing_consent`는 §5-1 선택 동의의 저장처(철회 시 false + 일시 갱신).
- **업로드 이미지**: base64가 jsonb에 포함되면 행이 수 MB — **Vercel Blob으로 분리**하고 data에는 URL만. (가입 시점에 base64 → Blob 업로드 일괄 변환)
- **게스트 병합 규칙**: 로그인 시 로컬 보드와 서버 보드가 둘 다 있으면 —
  - 한쪽이 빈 보드면 다른 쪽 채택.
  - 둘 다 내용 있으면 `updated_at`(로컬은 lastVisitAt) 최신 쪽을 제안하되 사용자 선택 시트("이 기기의 보드 / 저장된 보드 어느 쪽을 쓸까?"). 자동 덮어쓰기 금지.
- **동기화**: 실시간 아님. `saveBoard()` 후 디바운스 업서트(온라인일 때). 충돌은 last-write-wins + 편집 기기 안내.
- **롤백**: 서버 저장 실패해도 localStorage가 항상 진실 원천 — 기존 오프라인 동작 무손상.

### 5-1. 가입 폼 동의 (v7.5 추가)

Google 로그인 직후 최초 1회 동의 화면:

| 구분 | 항목 | 강제 여부 |
|---|---|---|
| **필수** | 개인정보 수집·이용 동의 (Google sub·email·이름, §7 수집 최소화 범위) | 미동의 시 가입 불가 |
| **선택** | 마케팅 정보 수신: **"마인드/자기발견/자기성장/코칭 관련 정보를 받아보는 것에 동의합니다."** | **미체크로 가입 차단 불가** — 선택 동의를 가입 조건으로 걸 수 없음(정보통신망법상 분리 동의 원칙). 기본값 해제 상태 |

- 선택 동의 여부는 `users.marketing_consent` + `marketing_consent_at`에 기록. 마이페이지 시트(§4)에서 언제든 철회 가능.
- 수신 채널(이메일 등) 실제 발송 시스템은 별도 라운드 — 이 단계에서는 동의 수집·저장까지만.

## 6. 기술 스택 후보

- **인증**: Auth.js(NextAuth v5) + Google Provider — App Router 표준, 세션 JWT. (Clerk도 후보지만 단일 provider엔 과함)
- **DB**: Neon Postgres (Vercel Marketplace, 무료 티어) / 이미지: Vercel Blob.
- 예상 작업량: 인증+스키마+업서트 API+병합 시트+마이페이지 시트 = 별도 1라운드 규모.

## 7. 프라이버시·삭제권

- 수집 최소화: Google sub·email·이름만. 보드 내용은 민감 정보(꿈·재정·관계)이므로 **분석 이벤트에 본문 절대 미포함** (현 track()도 id만 보냄 — 유지).
- 계정 삭제 = users·boards·Blob 즉시 삭제 + localStorage 유지 여부 선택.
- 개인정보처리방침 페이지 필요(가입 도입 시점에).

## 8. 결론

- **R2에서 구현** (v7.4 채택 → v7.5 유도 구조 확정). 첫 슬라이스:
  1. Auth.js(NextAuth v5) + Google Provider + Neon Postgres 스키마(§5) + 보드 1개 업서트 API + base64→Blob 변환.
  2. **온보딩 종료 선택 화면**(§2-1: Google로 시작 / 게스트로 시작 + 유의 카피).
  3. **첫 사진 직후 소프트 게이트 시트**(§2 B) + 대시보드 배너 재유도 + 배경화면 저장 시트 보조 유도(C).
  4. 동의 화면(§5-1 필수+선택) + 병합 시트(§5) + 계정 시트(§4 1차) + 개인정보처리방침 페이지(§7).
- 마이페이지 확장·다중 보드·마케팅 발송 시스템은 그 다음 라운드.

## 9. 운영 가이드 — 가입자·활동 정보 확인 (v7.5 추가)

가입 도입 후, 별도 관리자 페이지 없이 무료 도구만으로 확인하는 방법:

### Neon 콘솔 (가입자·보드 활동)
Vercel 대시보드 → Storage → Neon → "Open in Neon Console" → SQL Editor에서:

```sql
-- 전체 가입자 수 / 마케팅 동의자 수
SELECT count(*) AS total,
       count(*) FILTER (WHERE marketing_consent) AS marketing_ok
FROM users;

-- 최근 가입 20명 (이메일·가입일·마케팅 동의)
SELECT email, name, marketing_consent, created_at
FROM users ORDER BY created_at DESC LIMIT 20;

-- 최근 활동 유저 — 보드가 최근 7일 내 갱신된 계정
SELECT u.email, b.updated_at, b.schema_version
FROM boards b JOIN users u ON u.id = b.user_id
WHERE b.updated_at > now() - interval '7 days'
ORDER BY b.updated_at DESC;

-- 휴면 추정 — 30일 이상 보드 갱신 없는 가입자
SELECT u.email, b.updated_at
FROM users u LEFT JOIN boards b ON b.user_id = u.id
WHERE b.updated_at IS NULL OR b.updated_at < now() - interval '30 days';
```

### Vercel Web Analytics (익명 트래픽·행동)
- Vercel 대시보드 → 프로젝트 → Analytics 탭: 페이지뷰·방문자·경로별 유입.
- 코드에 심어둔 `track()` 커스텀 이벤트 5종(photo_first_entry, collage_view 등)은 **Pro 플랜 전용** — Hobby에서는 무해한 no-op(수집 안 됨). 이벤트 단위 활동 분석이 필요해지면 Pro 전환 or Neon 쿼리로 대체.
- 원칙 유지(§7): 분석 이벤트에 보드 본문 절대 미포함 — 유저·보드 식별은 Neon, 행동 집계는 Analytics로 분리.
